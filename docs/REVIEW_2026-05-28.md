# AWSomeQuiz — Code & UX Review (2026-05-28)

Scope: end-to-end audit of the deployed app at <https://awsomequiz.streamlit.app/>.
Sources reviewed: full `app/*`, `pages/*`, `streamlit_app.py`, supabase migrations 0001–0009, `app/styles.py`, `tests/test_imports.py`, plus a live Playwright walk-through (Home / Practice / Stats / Glossary / direct-URL deeplinks).

This is a recommendations document. Nothing in here has been implemented. Each item carries a severity (🔴 high / 🟡 medium / 🟢 low / 💡 idea) so you can decide where to spend.

---

## 0. Executive summary

The app is in good shape: clean module layout, well-named state keys, defensible patterns around RLS, cookies, and the timed-exam fragment timer. Phases 1–8 of the brief are shipped and stable. The headline pain points are:

1. 🔴 **Deeplinks for authenticated pages are broken on cold load.** `/practice`, `/stats`, etc. show *"Page not found. Running the app's main page."* until the user lands on `/` first. Caused by the cookie-restore happening *after* `st.navigation` decides which pages exist.
2. 🔴 **`pick_missed_question_ids` and `pick_weak_area_question_ids` do per-user full-scan client-side filtering.** Fine for one user, but every page render that touches them re-fetches up to 10,000 rows. Will become the largest cost driver as users practice.
3. 🟡 **The runner has no per-question reporting / notes UI**, only a Bookmark toggle. `question_reports` table exists but nothing writes to it from the UI.
4. 🟡 **No real tests** beyond import-smoke. Business logic (record_answer, complete_session, pick_*, streak math) is wholly untested.
5. 🟡 **Stats dashboard mixes "score history" with timed-only, but doesn't show the practice/review trend** — most engaged users will rarely take timed exams.
6. 🟡 **Domain weights are stored but not used for question selection.** A "timed exam" today returns 65 uniformly-sampled questions, not 16 Cloud Concepts + 19 Security + 22 Tech + 8 Billing as the real CLF-C02 exam does.
7. 🟡 **No spaced-repetition scheduling** on the missed/weak queues; questions reappear at random.
8. 🟢 Many small polish items below (mobile layout, keyboard nav, accessibility, etc.).

## 1. Bugs / functional issues

### 1.1 Deeplinks for authenticated pages 🔴

**Repro**: paste `https://awsomequiz.streamlit.app/practice` into a fresh browser tab (cookies intact). Streamlit shows "Page not found. Running the app's main page." and the URL silently rewrites to `/`. Verified via Playwright at `/practice` and `/stats`.

**Root cause** (in [streamlit_app.py:201-227](streamlit_app.py#L201-L227)):
```
restore_session_from_cookie()    # async — needs cookie_reader round-trip
session = get_session()
if session:
    pages = [Home, Practice, Stats, ...]
else:
    pages = [Login, ResetPassword, GuestPractice, Glossary]    # no /practice
pg = st.navigation(pages, position="sidebar" if session else "hidden")
```

The cookie_reader sends its value via a component message. The *first* run hits this code with `session = None`, so `/practice` isn't a registered page. Streamlit emits "Page not found" and renders the unauth default page. The cookie value arrives → rerun → now `/practice` IS registered, but the URL is already `/`.

**Fix options** (don't pick yet — happy to discuss):
- Always register the authenticated pages, gate them in-page with `if not current_user(): st.switch_page("login")`. Lets `st.navigation` resolve the URL on the first pass.
- Render a `st.spinner("Restoring session…")` block while `_read_refresh_cookie()` is in flight, then rerun.
- Skip `st.navigation` entirely and route by `st.query_params` ourselves.

### 1.2 Timed-exam loses per-question timing 🟡

[pages/timed_exam.py:365](pages/timed_exam.py#L365) passes `time_spent_seconds=0` to `record_answer`. The free-practice runner [app/components/runner.py:103](app/components/runner.py#L103) captures elapsed correctly via `qst_*` session keys. Timed mode could do the same — set the start timestamp the first time the question is displayed and compute on save. Useful for the "questions you spend longest on" insight down the line.

### 1.3 Auto-submit + `st.fragment` race window 🟡

[pages/timed_exam.py:84-97](pages/timed_exam.py#L84-L97): the timer fragment re-runs every 1 s. When `remaining <= 0`, `_finalize` is called from inside the fragment, which calls `st.rerun(scope="app")`. The `TIMED_AUTO_SUBMITTED_KEY` guard prevents double-finalize, but `complete_session` reads `user_answers` and updates `exam_sessions`. If the user happens to be saving an answer in the same second, two writers contend. Low impact (UPSERT is safe; trigger is idempotent), but the race exists. Consider stamping `completed_at` only via a DB-side `select … for update` or a guarded `UPDATE … WHERE completed_at IS NULL`.

### 1.4 `set_current_certification` race / stale cache 🟡

The sidebar cert picker [streamlit_app.py:60-70](streamlit_app.py#L60-L70) calls `set_current_certification(chosen) + st.rerun()`. But `_render_sidebar_mini_stats` is called *before* the picker each rerun, so the first render after a cert switch shows stats for the OLD cert. The comment in [app/queries.py:118-122](app/queries.py#L118-L122) acknowledges this. Trivially noticeable on a switch.

**Fix**: have the picker write to session_state, then `st.rerun()` *before* `_render_sidebar_mini_stats` runs (re-order; or move the picker above mini-stats in the layout flow).

### 1.5 Resume sets index = end when all answered 🟡

[pages/practice.py:67-68](pages/practice.py#L67-L68) and [pages/review.py:82-83](pages/review.py#L82-L83): if every question in the resumed session was already answered (e.g. user closed the tab after answering the last one), the loop falls through, the `else` sets `INDEX_KEY` past the end. The runner then immediately calls `complete_session`. Functionally correct, but the user gets no warning that "you actually finished it last time" — they expect to revise answers. Show a "session was complete — view summary" affordance instead.

### 1.6 Bookmarks page has no cert info shown 🟢

[pages/bookmarks.py:103-115](pages/bookmarks.py#L103-L115) shows stem preview + bookmark date but no domain / cert / question_id / difficulty. If a user has bookmarks for both CLF-C02 and DVA-C02 (which the current code filters to current cert anyway), switching the active cert hides bookmarks for the other one with no hint. Add a "you have X bookmarks in DVA-C02 (switch cert to see them)" line.

### 1.7 Account "Username" field is dead 🟢

[pages/account.py:40](pages/account.py#L40) saves `profiles.username` but it isn't displayed anywhere else (sidebar shows email). Either ship a leaderboard / profile page that uses it, or remove the field to avoid confusion.

### 1.8 `apply_session_to_client()` called twice per render 🟢

[streamlit_app.py:203](streamlit_app.py#L203) calls it once, then every page calls it again at module top. Harmless duplication but the `set_session()` call hits the supabase-py client; on a slow client this is a small extra latency per page.

### 1.9 Free Tier comment leaks Inbucket URL 🟢

[pages/login.py:121](pages/login.py#L121) "Locally: http://localhost:54324 -- Inbucket." appears in the success message *in production* too. Guard with `os.environ.get("STREAMLIT_RUNTIME_ENV")` or hide when `site_url()` isn't localhost.

### 1.10 `requirements.txt` / `pyproject.toml` drift 🟢

Two dep manifests means they can desync. The CLAUDE.md note "keep both in sync" is the only mechanism. Either generate `requirements.txt` from `uv export` in CI, or remove `requirements.txt` and let Streamlit Cloud read `pyproject.toml` directly (it can, as of mid-2025).

### 1.11 GitHub OAuth button always shows 🟢

Documented in code [pages/login.py:1-7](pages/login.py#L1-L7); not a bug, but a real "click and get a 400" UX hole if GitHub provider isn't enabled. Add a `gh_oauth_enabled = st.secrets.get("AUTH_GITHUB_ENABLED", "false")` toggle.

### 1.12 Question-report UI doesn't exist 🟡

`app/queries.py:report_question(...)` is defined and `REPORT_REASONS` is set, but no page or runner widget calls it. The brief mentioned "Report incorrect question" — should land as a `@st.dialog` button next to the Bookmark button in the runner.

## 2. Performance / scalability

### 2.1 `pick_missed_question_ids` does a full-table client filter 🔴

[app/queries.py:471-510](app/queries.py#L471-L510):
- fetches all `exam_sessions` ids for the user (10k cap),
- fetches all `user_answers` where session_id IN (...) AND is_correct=false (10k cap),
- builds a set, filters to current cert client-side.

For an active user with 100 sessions × 65 questions = 6,500 answer rows. Loads on every render of the Review page. The "Available" count caption at the top re-runs the pick.

**Fix**: SQL view `vw_missed_questions(user_id, certification_id, question_id)` materialised by the existing `user_answers_update_stats` trigger, indexed on `(user_id, certification_id)`. Or push the filter into the request: PostgREST has `eq()` chains; an RPC function `get_missed_question_ids(p_user, p_cert)` would be O(rows for this user) on the server.

### 2.2 `pick_weak_area_question_ids` rebuilds the full picture every call 🟡

[app/queries.py:424-468](app/queries.py#L424-L468) loads `_cert_question_ids` (up to ~915 today, ~5–10k future), loads all `question_stats` for the user, then loops. Cache this for ~30 s with `@st.cache_data` the same way `get_user_stats_summary` does.

### 2.3 `list_flashcard_decks` makes one PostgREST query per deck 🟡

[app/queries.py:710-730](app/queries.py#L710-L730) loops decks and runs `select count(*) where deck_id = …`. For 5 decks: 6 round-trips total (1 list + 5 counts). PostgREST has `?select=*,flashcards(count)` syntax that returns nested aggregations in a single request.

### 2.4 `get_practice_streak` reads 2000 sessions 🟢

[app/queries.py:847-878](app/queries.py#L847-L878). Fine today; cap should drop to ~365 (one row per day max). Add `.limit(365)`.

### 2.5 Magic-number `.limit(10000)` everywhere 🟢

15+ occurrences. Define `MAX_ROW_LIMIT = 10_000` in `app/queries.py` and use it. When you cross the cap (a power-user with >10k answers), silent truncation will mis-report stats.

### 2.6 `get_question_with_options` has TTL=5min 🟢

Per-question read. Cache hit rate is very high once a user is mid-session, but if they bookmark then re-open the runner, the cache is keyed on `question_id` so it survives. Could bump to 1 h since question content is effectively immutable.

### 2.7 Sidebar mini-stats fires three DB calls per render 🟢

[streamlit_app.py:80-89](streamlit_app.py#L80-L89) calls `get_current_certification` (cached), `get_user_stats_summary` (cached 30s), `get_practice_streak` (cached 30s). Total ~3 cached lookups + 0–3 DB queries depending on cache state. Fine for now; if it becomes hot, a single `get_sidebar_bundle(user_id)` RPC would halve it.

## 3. UX / product

### 3.1 Domain-weighted timed exam 🟡

The real CLF-C02 exam is **24/30/34/12** across its four domains. `start_timed_session` currently picks 65 questions uniformly at random. A "true" exam simulation should pick **~16 Cloud Concepts + ~19 Security + ~22 Tech + ~8 Billing**, picked at random within each domain. The `domains.weight` column already holds this; the existing `pick_question_ids(domain_ids=[d])` can be called per-domain. DVA-C02's weights are 32/26/24/18.

This is the single highest-impact UX change for cert-prep utility.

### 3.2 Stats: practice trend, not just timed 🟡

[pages/stats.py:97-117](pages/stats.py#L97-L117) plots only timed-exam scores. Users who study via short practice sessions never see a trend line. Add a second chart: rolling-30-day accuracy (or rolling-100-question accuracy) computed from `user_answers`. Bonus: per-domain trend lines.

### 3.3 Spaced repetition for missed / weak 🟡

Today: `pick_missed_question_ids` is uniformly random over the wrong set. SM-2 / FSRS gives way better retention. The DDL already supports it — add `next_review_at TIMESTAMPTZ`, `ease_factor REAL`, `interval_days INT` to `question_stats` (flashcards already have these conceptually via reviews). Initial implementation: SM-2 on review correctness from `record_answer`. Surface "X questions due today" on Home.

### 3.4 Per-question Notes / Tags 🟡

A bookmark is binary. Many users want to scribble *why* they got a question wrong ("confused S3 lifecycle with Glacier expiration"). Add `bookmarks.note` (already there!) editing UI, and a `bookmarks.tags TEXT[]` column for user-defined buckets.

### 3.5 Report-issue dialog 🟡

See 1.12. Should be a 1-click button next to Bookmark on every question; opens a `@st.dialog` with the 5 existing reasons + free-text "details" field. Posts to `question_reports`. Add an admin view (or just a CSV export script for the maintainer) to triage.

### 3.6 In-app "Study plan" 💡

Pick a target date and target score, get a per-day goal of N questions and M flashcards. Surface on Home: "Day 14/30 — 47 questions today to stay on track". This is the killer feature for prep apps; no AI required, just arithmetic.

### 3.7 Onboarding tour 💡

First-time signed-in user lands on a Home page with no context. A 4-step in-app tour (`st.popover` or a manual overlay) covering: pick a cert, run your first practice, review your missed, check stats — drops abandonment.

### 3.8 Glossary deep-linking from question explanations 💡

Per-option `explanation_detailed` HTML often mentions terms like "S3 Glacier Deep Archive" — link these tokens to `/glossary?q=Deep+Archive` so the user can click through and back. Implementation: post-process the explanation HTML against the glossary index to inject anchor links.

### 3.9 Keyboard shortcuts 💡

The brief deferred this. Real exam-prep power users want `1/2/3/4` to select options, `Enter` to submit, `n` for next, `b` for bookmark. `streamlit-shortcuts` (free, ~3KB) makes this 4 lines.

### 3.10 Mobile responsiveness 🟡

The timed-exam sidebar grid uses 5 columns of question buttons. On a phone, the sidebar is fullscreen (good) but the 65-button grid wraps awkwardly. Test on `iPhone 15 Pro` (390×844) and `Galaxy S24` (360×800) — both bend the layout. Also the dark-mode toggle at top-right gets pushed below the title on narrow screens.

### 3.11 No autosave indicator 🟢

Both runners save on every submit, but there's no visual confirmation. A tiny "Saved" pulse next to the question counter would reassure.

### 3.12 No celebratory feedback on pass 🟢

[pages/timed_exam.py:127](pages/timed_exam.py#L127) shows a `st.success("PASSED — 75%")` banner. Could add `st.balloons()` or `st.snow()` on first pass per cert. Tiny effort, high delight.

### 3.13 Quit/abandon UX should be confirmed 🟢

Both runners have a "Quit session" button. Clicking immediately calls `abandon_session` and reruns — no confirmation. Wrap in `@st.dialog` ("Quit and lose progress? Your answers so far are kept in the session record but you can't resume").

### 3.14 No question difficulty / time-to-answer surfaced 🟢

Schema has `time_spent_seconds`. Could compute per-question average time across users (or just the active user) and surface "median solve time: 38s" on the review screen.

### 3.15 No exam-history detail page 🟢

Stats shows recent-sessions table but rows aren't clickable. Add `pages/session_detail.py` that takes `?id=…` and shows the full review of that exam (existing `get_review_bundle` already does the heavy lifting).

### 3.16 No share-result feature 💡

Twitter-style "I scored 87% on the AWS CLF-C02 practice — try it!" with an image. Increases organic acquisition. Tricky on Streamlit since rendering images server-side is heavy, but a static svg with the score baked in is fine.

### 3.17 Email-confirmation message doesn't mention spam 🟢

[pages/login.py:121-122](pages/login.py#L121-L122) tells local users about Inbucket. Production users: add "check spam folder" line.

### 3.18 Account page lacks "Delete my account" 🟡

GDPR-aware users expect this. Should soft-delete the auth user (Supabase admin API) + cascade-delete or anonymise `user_answers`, `bookmarks`, `question_stats`, `flashcard_reviews`. Even a manual "email us to delete" link is better than nothing.

### 3.19 No display-name preference 🟢

Email is shown in the welcome line and sidebar. Users may not want their email visible at "AWSomeQuiz, [me@company.com]" while screen-sharing. Use `profiles.username` (already in DB) as the display name once set.

## 4. Code quality

### 4.1 Sparse tests 🟡

`tests/test_imports.py` does py_compile + import. Nothing tests business logic. Suggested additions (in priority order):
1. `tests/test_session.py` — mock `app.db.get_supabase`, verify `record_answer`, `complete_session`, `_finalize`, abandon flow.
2. `tests/test_queries.py` — `pick_*` selectors with fixture data, `get_practice_streak` with 0/1/multiple days, `get_per_domain_accuracy` with untagged + tagged.
3. `tests/test_auth.py` — cookie save/delete queue mechanics, refresh-failure path.
4. `tests/test_glossary.py` — JSON schema validation, no duplicate `term`, all `category` values in allowed set, all `certs` values in allowed set.

### 4.2 No CI 🟡

`uv run pytest` works locally but nothing enforces it. Add `.github/workflows/ci.yml` that runs `ruff check`, `ruff format --check`, `pytest`, on every push and PR. ~30 line YAML.

### 4.3 Broad `except Exception` everywhere 🟢

`# noqa: BLE001` shows up in 10+ places. Most have legitimate "swallow + show user-facing error" intent. Acceptable, but consider narrowing the few that catch specifically `gotrue.errors.AuthError` or `httpx.HTTPError` for cleaner stack traces in Sentry-equivalent logging.

### 4.4 Type-check not enforced 🟢

Type hints are present everywhere. Add `pyright` or `mypy --strict` to CI; fix the resulting drift. Two hours of work, lifetime of value.

### 4.5 Mixed import style 🟢

`from app.session import ...` mostly, but two functions in `app/auth.py` and `app/queries.py` do `from app.queries import …` *inside* the function body to dodge circular imports. Pulling all the per-cert helpers out into `app/cert.py` would let the auth.py / queries.py boundary be cleaner.

### 4.6 No structured logging 🟢

`print()`-level debugging would never have caught the cookie/header issue you debugged earlier. Add `import logging; logger = logging.getLogger(__name__)` at module top, log at `INFO` for auth flow + session lifecycle, `WARNING` for swallowed exceptions. Streamlit Cloud captures stdout; you'll thank yourself.

### 4.7 Glossary JSON gets large (382 entries × ~500 char definitions ≈ 200 KB) 🟢

Still tiny by web standards, but loaded on every Glossary page hit (cached via `@st.cache_data(ttl=300)`). At ~1000+ entries consider splitting per-letter and loading lazily. Or move the glossary into Postgres with a real FTS index (and drop the cache).

### 4.8 No security headers 🟢

Streamlit Cloud doesn't let you set CSP / X-Frame-Options. Not blocking; flag in case you self-host.

### 4.9 Cookies — single function does multiple things 🟢

`app/cookies.py` is correct but `read_cookie_client_side` has a key argument that's tied to the custom-component instance. If two unrelated places need the same cookie they'll fight. Wrap as `get_refresh_token()` / `get_dark_mode_pref()` so callers don't pass `key=` strings.

### 4.10 No `Makefile` target for the glossary helpers 🟢

`scripts/_glossary_gap_report.py` and `_glossary_merge.py` are used during glossary expansion. Document in `docs/how_to_run.md` ("How to add more glossary entries").

## 5. Security / compliance

### 5.1 RLS is on, but `question_reports` is insert-only without rate limiting 🟡

A logged-in user can spam thousands of `question_reports` rows. Schema doesn't prevent it. Add either:
- Postgres trigger that rate-limits inserts per user per minute.
- Per-user limit in app code before insert.

### 5.2 Service-role key handling 🟢

`docs/how_to_run.md` covers this, but verify the deployed Streamlit Cloud doesn't have the service-role key in secrets (it shouldn't — `app/db.py` only reads anon key).

### 5.3 Email-rotation flow 🟢

Supabase supports `email_change` OTP. Currently not exposed in the UI. Low priority but worth a stub on Account.

### 5.4 No CSRF on logout button 🟢

Streamlit's reruns aren't HTTP requests in the traditional sense, so CSRF is N/A. If you ever expose a custom HTTP endpoint (e.g. for a webhook), revisit.

### 5.5 PKCE flow ✅

Already enabled in `app/db.py`. Good.

## 6. Content / data

### 6.1 Domain tagging incomplete for some questions 🟡

Per CLAUDE.md, `_classify_clf_questions.py` and `_classify_dva_questions.py` ran heuristically. Stats `get_per_domain_accuracy` already handles "Untagged" gracefully. But if 5–10% of questions are mis-tagged (because keyword matching is fuzzy), per-domain accuracy is unreliable.

**Recommendation**: build a small admin page (or just a CLI script) that lets the maintainer audit `is_active=true` questions with `domain_id IS NULL` *and* the ones the classifier marked with low confidence, and bulk-update from the terminal. Or: add a Claude/Haiku one-shot using the question text as input — model emits domain code, you verify by hand.

### 6.2 Question quality — Q370 disabled, others? 🟢

Q370 was marked `is_active=false` because options were truncated. Are there other corrupted rows? Worth a quick:
```
select id, external_id from questions
where (select count(*) from options o where o.question_id = questions.id and o.is_correct) = 0;
```
to find questions with no correct option. If any others come back, deactivate them too.

### 6.3 Flashcard reviews don't update `flashcard_stats` consistently 🟢

Trigger maintains it via insert on `flashcard_reviews`. Verify the trigger fires on every review and the `last_correct_at` field updates only when `knew_it=true`. (Test: a deck of 5, do 5 wrong then 5 right, see "known" go to 5.)

### 6.4 Glossary covers CLF-C02 + DVA-C02 only 🟢

Once you add SAA / SAP / SOA question banks (per CLAUDE.md plan), the glossary expansion process needs another batch run. The `_glossary_gap_report.py` script generates the list automatically — that's the right workflow.

## 7. Specific Playwright observations

These came up during the walk-through; some overlap with above.

- **Console errors on `/glossary`**: two 404s for `_stcore/health` and `_stcore/host-config` URLs that include the `/glossary` prefix. Streamlit Cloud routing artifacts. Cosmetic; ignore.
- **No accessibility labels** on the dark-mode toggle (just "Dark mode"); fine, but a `screen reader` user without context might guess wrong.
- **Question buttons in timed-exam sidebar** lack `aria-label` distinguishing "answered" vs "unanswered" — only visual "v " prefix indicates state.
- **Sidebar mini-stats** displays "ACCURACY 66.7% / STREAK 2d / SEEN 6" — labels are tight; on narrow screens this clips. Test mobile width.
- **Home page hero**: shows "Q" without a thousands separator (915 reads fine; 9,150 would not). Already future-proofed by using `q_count` directly.
- **Practice cert dropdown** (sidebar) — when only one cert is present it renders as static text. When two are present it renders as a selectbox. Good.
